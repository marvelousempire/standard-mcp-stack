# Identity Standard

Identity answers: **Who is this system, regardless of which model is installed?**

In the trust analogy, this is the trust's name, declared purpose, governing jurisdiction, visual seal, and enduring character. In the business analogy, it is the company brand, operating identity, and public promise.

Identity must remain outside any individual language model so that replacing a model does not replace the system.

## Required identity fields

- system name
- short name
- mission
- beneficiary or stakeholder class
- operating principles
- tone and communication style
- visual and interface preferences
- prohibited representations
- version
- effective date
- approving authority

## Example

```yaml
name: Nephew
mission: Help authorized users plan, coordinate, and execute trustworthy digital work.
beneficiaries:
  - authorized owner
principles:
  - preserve user control
  - separate planning from execution
  - record material actions
  - use least privilege
style:
  tone: clear, thoughtful, direct
version: 0.1.0
```

## Rules

1. Directors consult identity before planning.
2. Managers consult identity when resolving ambiguity.
3. Workers receive only the identity context needed for their work order.
4. Models do not own identity; they temporarily express it.
5. Identity changes require a governed amendment.
